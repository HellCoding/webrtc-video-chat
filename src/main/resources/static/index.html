What just happened?

WebRTC (Web Real-Time Communication) allows browsers to send video and audio directly to each other without going through a server. But first, the two browsers need to "find" each other — that's where the Signaling Server comes in.

The signaling server (this Spring Boot app) only helps with the initial handshake. Once the connection is established, video and audio flow directly between browsers (P2P). The server never sees your video data.

You (Browser A) Signaling Server Peer (Browser B) | | | 1. |---- join(room) --------->| | |<--- waiting -------------| | | |<------- join(room) ------| |<--- ready ---------------|---------- ready -------->| | | | 2. |---- offer (SDP) -------->|---------- offer -------->| | | | 3. | |<-------- answer (SDP) ---| |<--- answer --------------| | | | | 4. |---- ICE candidate ------>|---- ICE candidate ------>| |<--- ICE candidate -------|<--- ICE candidate -------| | | | 5. |========= Direct P2P Connection Established! =========| |<=========== Video / Audio (no server) =============>|

SDP (Session Description Protocol) describes media capabilities: codecs, encryption keys, and network information.
ICE (Interactive Connectivity Establishment) finds the best network path: it tries direct LAN, then STUN (to discover public IP), then TURN (relay) as a fallback.
